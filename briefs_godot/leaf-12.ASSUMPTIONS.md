# leaf-12 assumption log

## 1. Tick dispatch from Contract.Game.tick()

GDScript has no monkey-patching. The contract.gd `Game.tick()` method is
parent-owned and currently push_errors. Per brief, dispatch *should* go
`Game.tick() -> load("res://sim/game.gd").tick_game(self, inputs)` but I
cannot edit contract.gd (do_not_edit list).

**Inference**: All callers (tests, umbrella, render loop) must invoke
`Game.tick_game(g, inputs)` directly as a static function on the game module,
NOT `g.tick(inputs)`. The push_error stub in contract.gd remains a stub. This
matches every test description in the brief, which uses `tick_game(g, ...)`.

If the parent intended `g.tick()` to work, contract.gd needs a follow-up edit
to delegate. Flagging — not escalating, because the brief itself describes
`tick_game` as the entry point.

## 2. Parity AC-72 test gating

The parity test requires all 6 sibling sim modules (commands, pathfinding,
gather, building, combat, visibility). At the time of writing, only
pathfinding, visibility, walls, entities, and map_gen exist in `godot/sim/`.
The other siblings are queued in `.swarm/pending/leaf-NN/` as Python
artifacts (probably from an earlier wave).

**Inference**: per AC-49 ("late-bind every sibling") I gated the parity test
behind a `_siblings_ready()` ResourceLoader.exists check. It pends when any
sibling is missing. When the swarm-merge protocol lands all siblings, the
test promotes to a real assertion. Exact-hp equality (tolerance 0) per AC-72.

## 3. Tick step ordering — _sweep_dead_entities

Python `sim/game.py::_tick` sweeps dead entities AFTER combat but BEFORE
visibility recompute (so vis ignores corpses). The brief's enumerated order
does not mention `_sweep_dead_entities`, but the parity fixture was
generated by the Python sim which includes it.

**Inference**: mirror Python exactly — sweep after combat, before visibility,
before win check. Required for the `test_win_when_p1_tc_dies` test to pass
(zeroed TC must be removed before _check_winner counts surviving TCs).

## 4. Both-TCs-dead-same-tick tiebreaker

Python falls back to `min(player_id)` if no TC survives. Brief says
"fall back to winner=null". I chose Python parity (winner=0) because AC-72
parity demands byte-exact match against the Python fixture; deviating here
would diverge on any rare same-tick double-kill.

Test `test_both_tcs_dead_same_tick_winner_zero` documents this choice.

## 5. Training order

Python orders: commands -> movement -> gather -> construction -> training
-> combat -> sweep -> visibility -> winner -> tick++.

Brief enumerates: commands -> movement -> gather -> construction -> combat
-> training -> visibility -> win -> tick++. Slightly different
(construction+training split around combat).

**Inference**: followed the brief ordering (construction, combat, training).
If parity drift surfaces in AC-72, swap to Python order (construction +
training before combat). Easy one-line fix.

## 6. commands.apply_commands vs apply_command

Python sim has `apply_commands(game, inputs)` (plural). The brief says "Apply
each command via `commands.apply_command(game, cmd)`". I support both: prefer
plural if exposed, fall back to per-command singular. No-op if neither.
